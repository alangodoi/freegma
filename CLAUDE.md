# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `bun install` — install dev deps (only `@types/bun`)
- `bun start` / `bun run server.ts` — start the editor at http://localhost:3001
- `bun build --target=browser public/editor.js --outdir /tmp/out` — quick syntax check of the client bundle. There is no build step in production; `public/` is served as-is.

There are no tests, linter, or formatter configured.

## Architecture

Two moving parts:

1. **`server.ts`** (~50 lines) — a minimal Bun HTTP server. It serves files under `public/` statically and exposes **one** API route, `GET /api/defaults`, which reads every `.svg` in `./icons/` and returns `{ name: svgMarkup }`. There is **no database and no per-user/per-drawing persistence on the server** — this was intentionally removed. Do not reintroduce a DB or a `/api/icons` POST/DELETE route without asking.

2. **`public/editor.js`** (~1300 lines, single ES module) — the entire editor. All state lives in-memory in the browser. No bundler; loaded directly via `<script type="module">`.

### Client state model

- `drawings: Record<string, { svg: string, width: number, height: number }>` — the session's drawings keyed by name. Each drawing stores its own canvas size (Figma-style frames).
- `currentId`, `currentW`, `currentH` — the active drawing and its dimensions.
- `selection: SVGElement[]` — selected shapes.
- `undoStack: string[]` — up to 50 serialized `svgCanvas.innerHTML` snapshots.
- `vbX/vbY/vbW/vbH` — editor viewBox (pan+zoom state), recomputed from `currentW/H + 10% padding` on load.

The `<svg id="svgCanvas">` is the single source of truth for the current drawing. It always contains the shape elements plus two appended helpers: `boundsRect` (the dashed frame outline) and `handlesGroup` (selection box + resize/rotate handles). Both carry `data-bounds`/`data-handles` attrs so `cleanClone()` can strip them on export.

### Persistence flow

Fully client-side: **Download** serializes `svgCanvas` via `cleanClone()` (which rewrites `viewBox`/`width`/`height` to the drawing's own size) and triggers a browser download. **Import** takes a file picker or pasted markup; `extractSize()` parses `viewBox` or `width`/`height` to recover per-drawing dimensions. Reloading the page wipes session state and reloads the `./icons/` starter templates into the sidebar list.

### Rect-path (per-corner radii)

SVG `<rect>` only supports uniform `rx`/`ry`. To support Figma-style per-corner radii, `setRectLike()` transparently converts a `<rect>` to a `<path data-rect="1">` whose geometry lives on `data-x/y/w/h/tl/tr/br/bl` and whose `d` is regenerated by `renderRectPath()` / `roundedRectD()`. If all four corners become equal again it converts back to a `<rect>`. When this swap happens the element reference in `selection[]` must be updated via `swapSelected(oldEl, newEl)`, then `populateProps(newEl)` must re-render. `moveElement()`, `resizeElement()`, and `scaleElement()` each have a dedicated branch for `path[data-rect="1"]` so dragging/scaling updates the `data-*` attributes and re-renders the `d`, rather than appending `transform` operations.

### Scaling the whole drawing

`scaleDrawing(newW, newH)` recomputes `sx`/`sy` from `currentW`/`currentH`, then calls `scaleElement()` on every child. `scaleElement()` multiplies concrete coordinate attributes for primitive shapes, scales `stroke-width` by `min(sx, sy)`, rescales `translate()`/`rotate(pivot)` in any `transform` attr, and for free-form paths parses and scales every command's numeric operands via `scalePathD()`. `scalePathD` handles M/L/T/H/V/C/S/Q/A in both cases and preserves A's three flag values.

### Tooltip system

Hints are delivered via a single `#hintTip` div appended to `<body>` and positioned with `position: fixed` via JS (`showHint()` in `editor.js`). It clamps to the viewport and flips above when near the bottom. Elements participate by setting `data-hint="..."`. Do not add per-element `title` attributes or CSS `::after` tooltips — they would clip under the canvas and sidebars.

### UI structure and design tokens

`public/style.css` defines CSS custom properties for colors, spacing (`--s1..s4`), control height (`--h`), and radius. Side panels use a consistent `.panel > .panel-head + .panel-body` pattern. Controls in the properties panel are built by `field(label)` and `miniInput(label, value, opts)` helpers in `editor.js`; prefer extending those rather than re-creating ad-hoc rows. The shape tool buttons use `.tool-grid > button > .ti + .tl` (icon glyph + small label).

### Things to be careful with

- Do **not** add elements to `svgCanvas` after `handlesGroup` — new shapes must be `insertBefore(el, handlesGroup)` so handles always draw on top.
- `cleanClone()` is the only correct way to serialize a drawing; it strips helper elements and sets the export viewBox to the drawing's own W×H.
- `pushUndo()` snapshots before mutations; any new mutation path should call it. `popUndo()` restores from the serialized HTML.
- Path `d` parsing in `scalePathD` relies on a regex that splits by command letters; new features that emit exotic path commands should be tested against it.
